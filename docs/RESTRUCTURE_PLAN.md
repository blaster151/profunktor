# Source Restructure Plan

## Folder rationales
- **src/arrow**: Arrow/Kleisli/CoKleisli abstractions and SF/Mealy related laws and implementations.
- **src/category**: General category-theoretic bridges and utilities that are not specific to bicategories or homotopy.
- **src/core**: Core kinds, typeclasses, natural transformations, small algebraic utilities. These are foundational and imported broadly.
- **src/data/either**: Either data type and related helpers live under data/either for discoverability and cohesion.
- **src/data/maybe**: Maybe/Option and their unified builders go under data/maybe as algebraic data types.
- **src/data/result**: Result/Ok/Err and their unified interfaces are grouped together under data/result.
- **src/effects**: Effect monads and IO/Task/State variants grouped as effect systems.
- **src/fluent**: Fluent API and related runtime installation kept in a dedicated namespace.
- **src/free**: Free/Cofree modules and related constructions clustered for recursion schemes and categorical encodings.
- **src/homotopy**: Homotopy/type-theory bridges and synthetic differential geometry under dedicated research domains.
- **src/lawvere**: Lawvere theories and comodel formulations co-located for algebraic effects work.
- **src/legacy**: Items not recognized by rules go to legacy for manual triage with minimal disruption.
- **src/math**: General math helpers and rational arithmetic grouped under math.
- **src/optics**: Profunctor optics core, instances, laws, and utilities belong under optics for a single entry point.
- **src/polynomial**: Polynomial functors, their algebras, monads, and categorical characterizations.
- **src/stream**: Stream core, fusion, state, and coordination modules unified under a dedicated stream namespace.
- **src/trees**: Tree utilities and canonical trees maintained under a trees namespace.
- **src/util**: Small utilities that donâ€™t fit elsewhere. These provide glue code and helpers used across modules. Keeping them in one place reduces incidental dependencies.

## Consolidation recommendations for prefix groups
- either*: Keep core type (`either.ts`) distinct. Consolidate `either-classes.ts`, `either-hkt.ts`, and `either-traversable.ts` behind a single public barrel (`data/either/index.ts`) and consider merging `either-ops-table.ts` into tests or doc-generated tables if it is not used at runtime. Keep `either-derivation.ts` only if compile-time helpers are genuinely reused; otherwise inline into call sites. `either-register.ts` should export from the same index for a single import surface.
- maybe*/option*: Keep `maybe.ts` as the canonical type; move `option.ts` to re-export aliases or convert to thin wrappers to avoid dual entry points. Barrel-export `maybe-*` from `data/maybe/index.ts`. If `maybe-fluent.ts` is only fluent glue, fold into the fluent layer and re-export from the barrel.
- result*: Prefer a single `result.ts` with constructors and pattern-matching; fold `result-unified.ts` into the same module (or re-export from `index.ts`) unless there are materially different semantics. Keep a single public path `data/result`.
- optics*: Create subpackages inside `src/optics`: `core/` (core, adapter, instances, traversal), `laws/` (law harness), `dialects/` (dialens, affine, plens, preoptic), `algorithms/` (scheduler), and `bazaar/` (all bazaar-* files). Remove duplicate copies: `optics-core - Copy.ts`, `optics-indexed - Copy.ts`, `optics-scheduler - Copy.ts`. Provide a top-level `optics/index.ts` that re-exports curated entry points to shrink import surface.
- stream*: Introduce `src/stream/core/` (types, core, pull, resource), `src/stream/ops/` (ops, fusion, boundaries, combinator-roles), and `src/stream/adapters/` (observable-lite, observable-optics). Where feasible merge small glue files (e.g., `stream-boundaries.ts`) into `ops` to reduce fragmentation. Add a `stream/index.ts` barrel.
- free/cofree*: Group all `cofree-*` under `free/cofree/` and merge the multiple lazy variants (`cofree-lazy.ts`, `cofree-lazy-iter.ts`, `cofree-lazy-para.ts`, `cofree-lazy-bfs-build.ts`) into a single module with named exports for strategies. Keep `free-applicative.ts`, `free-monad-module.ts`, and `free.ts` but export via `free/index.ts`.
- arrow*: Rename `arrows-*` files to drop the plural and relocate to subfolders: `arrow/kleisli/` and `arrow/cokleisli/`. Consolidate law glue (`arrow-laws-glue.ts`) with the general laws runner or expose via `arrow/laws/index.ts`. Keep `mealy` and `mealy-cofree` under `arrow/machines/` with a barrel.
- category/adjunction*: Create `category/adjunction/` and move `adjunction.ts`, `adjunction-core.ts`, `adjunction-framework.ts`, `adjunction-free.ts`, `adjunction-registry.ts` there with an `index.ts` to present a single import path.
- polynomial*: Keep `polynomial-functors.ts` as the central entry and move related modules to subfolders: `theory/` (characterizations, degree, topology, 2category, calculus) with a barrel. The current split looks by design; prefer barrels over file merges here.
- lawvere*: Small set; keep separated for clarity but add `lawvere/index.ts` to re-export.
- core*: Keep modules separate (hkt, typeclasses, nat, semiring, monoids, readonly, immutable, trampoline, yoneda, partial). Provide a stable `core/index.ts` and consider folding very small helpers (e.g., `partial.ts`) into a single `core/primitives.ts` if import churn is high.
- homotopy*: Large research-oriented modules (khavkine-*, infinity-simplicial-*, topological-quantum-*) are appropriately split by topic; prefer barrels (e.g., `homotopy/index.ts`) over consolidation. Move any `*.examples.ts` into a dedicated `examples/` area if they are not required at runtime.
- legacy/* families (cooperad-*, deformation-*, dg-*, gadt-*, selective, normal-functors-slice, etc.): leave as-is for now. If actively used, promote to a stable namespace with barrels; if not, consider archiving or moving to a separate package.
